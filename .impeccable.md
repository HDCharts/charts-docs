## Design Context

### Users
Primary users are Kotlin Multiplatform engineers integrating charting into production apps. They are usually time-constrained and need to validate capability, copy working examples, and adapt styles quickly.

Secondary users are maintainers upgrading between versions who need migration clarity and API certainty with minimal ambiguity.

Core job to be done: move from "I need a chart" to "I shipped a correct chart" in minutes, with confidence.

### Brand Personality
Three-word personality: Precise, Reliable, Approachable.

Voice and tone should be technically rigorous, plainspoken, and calm. The interface should primarily evoke confidence and momentum, not hype.

Reference direction: documentation experiences with strong hierarchy and fast scanability (for example Kotlin, React, and modern product-doc patterns).

Anti-reference direction: marketing-heavy layouts, decorative visual noise, or playful styles that reduce technical trust.

### Aesthetic Direction
Dark-theme-only documentation experience. No light mode is planned or required.

Use a restrained technical palette: slate/neutral foundations with indigo + cyan as primary accents and warm tones reserved for warnings/critical signals.

Preserve code-first ergonomics: stable left navigation, clear section rhythm, predictable heading scale, and side-by-side demo/code where viewport size allows.

Motion should be subtle and purposeful. Favor short fades and state transitions; avoid ornamental animation.

### Design Principles
1. Optimize for copy/paste confidence first.
2. Favor information hierarchy over decoration.
3. Keep behavior and styling consistent across versions.
4. Treat accessibility as a default requirement, not a polish pass.
5. Prefer clarity and performance over visual novelty.
